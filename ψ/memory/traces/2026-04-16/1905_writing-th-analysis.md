## Summary: writing-th Evolution and Issues

### 1. Issues & Performance
*   **The "Contextual Drift"**: Early versions of the skill struggled with maintaining a consistent "Oracle" voice while trying to adopt specific report styles, often leading to hallucinations of style rules.
*   **Term Ambiguity**: The skill often included overly technical jargon (e.g., service blueprint, touchpoints) that alienated non-expert participants in workshops.
*   **Over-claiming**: A major issue was the agent claiming to have "fully materialized" a style pack when it had only referenced global resonance rules. This eroded human trust.
*   **Poor Question Design**: Early questions were often too broad, forcing participants to guess what was being asked instead of providing actionable input.

### 2. Improvements & Redesign
*   **Layered Style System (Option C)**: Redesigned the skill to use a hierarchy: (1) Project-specific example report as anchor → (2) Session Style Pack Summary → (3) Oracle Resonance (Safety Rails).
*   **Governance-led Condensation**: Style packs are no longer treated as "fully materialized" unless a deliberate condensation pass has inlined the specific rules into the project brief.
*   **Action-Oriented Questioning**: Questions are now structured as: (Context + Current Process) + (Specific Bottleneck) + (1-2 Actionable Suggestions).
*   **Modality Discipline**: Standardized Thai modality usage to differentiate between Foresight/Uncertainty (using อาจ/มีแนวโน้ม) and Evidence-based Analysis (using สะท้อน/ชี้ให้เห็น).
*   **Administrative Anchoring**: Shifted reporting logic from generic vulnerability/resilience frameworks to "Administrative Traces" (e.g., budget logs, LPA codes), making reports policy-relevant and implementable for Thai officials.
