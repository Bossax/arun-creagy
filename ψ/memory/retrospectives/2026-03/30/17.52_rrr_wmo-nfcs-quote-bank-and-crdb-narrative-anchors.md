## Honest Feedback (friction points)
1) NotebookLM MCP is fragile under real usage: authentication state can drift, and `ask_question` timeouts force unnatural micro-prompts. This adds cognitive overhead and encourages shallow extraction unless we deliberately persist.

2) Terminal output is often noisy or truncated (ANSI control characters, wrapping, partial previews). It makes it harder to do careful retrospection from command results alone, and it increases the chance of missing an important warning (e.g., line-ending warnings) unless we explicitly hunt for them.

3) Repo hygiene is becoming a recurring drag: CRLF/LF warnings, mixed uncommitted changes across unrelated areas, and “config/tooling” diffs interleaved with narrative artifacts all increase the cost of safe commits. The system works best when we keep a tight boundary, but the environment makes boundary-keeping effortful.

## Lessons Learned
- NotebookLM MCP extraction works best when the query is kept atomic (bounded length + one topic), especially when timeouts appear.
- For MCP auth issues, cleanup (while preserving library) + a clean browser session is usually faster than repeated retries.
- Commit boundary discipline is not “process overhead”; it materially reduces future confusion when multiple narrative streams (WMO-NFCS vs CRDB vs tooling) are active in the same workspace.
