## AI Diary (first-person)

I felt a very specific kind of tension in this session: the work was not “writing better Thai,” it was designing a system that can reliably *produce* better Thai across sessions without collapsing into ambiguity. The moment the example-report precedence became explicit, it clarified why the previous approach was brittle: a single, persistent resonance voice cannot fully cover sponsor-specific terminology and section flow. At the same time, leaning too hard on session-specific examples can cause style drift and accidental copying.

Option C (session style pack summary + optional promotion) gave a stable structure to that tension. It let the session be concrete—“this example report is the primary reference; here are the decisions we are making today”—while keeping the long-lived resonance files as safety rails. I also noticed how operational reliability shapes architecture: because `oracle_learn()` can sometimes say “success” without leaving a file on disk, the learning workflow must include a defensive verification step. That isn’t elegant, but it is honest.

The outcome feels like a more human-respecting system: the plan becomes the shared contract; the outline stop restores agency; and learning is a separate, explicit step rather than an implicit side effect.

## Honest Feedback

1) I repeatedly drifted into producing in-chat artifacts without immediately materializing them as files, which conflicts with the strict “document to remember” loop.
2) There is still complexity in coordinating four style layers (example report → session summary → resonance → learnings). The skill spec must remain explicit, or future sessions will reintroduce confusion.
3) Tool reliability (especially around Oracle learn materialization and vector-search availability) creates additional operational ceremony. It is manageable, but it needs to be treated as first-class workflow, not incidental debugging.

## Lessons Learned

- The correct “Oracle way” for writing style is layered: session-specific mimicry sits on top of durable resonance safety rails, with explicit recording in the writing plan.
- Learn-back must be a separate skill so it can be invoked from session closure without duplicating comparison logic.
- When tool outputs are nondeterministic, guardrails must be encoded into the skill spec (and not only into memory).
