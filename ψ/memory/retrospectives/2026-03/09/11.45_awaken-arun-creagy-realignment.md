## Honest Feedback

1) **Shell ambiguity created unnecessary churn.** Commands were initially written for PowerShell but actually executed in Git Bash. This caused parse errors and wasted time. Fix: assume Git Bash and use POSIX syntax (`date`, `mkdir -p`, pipes) consistently.

2) **Oracle skills update verification was confusing.** After installing v2.0.10 with `bunx`, the local `oracle-skills` binary still reported older skill versions (v1.5.79). This is likely an environment/agent activation nuance (restart required) or multiple installations. Fix: treat `bunx ... list -g` as source-of-truth for installed bundles, and record that the interactive agent must restart to load newly installed skills.

3) **“Fresh repo” assumption didn’t match reality.** This repo already contained extensive ψ content and git history. The ritual still works, but the session needed a deliberate “realignment” framing to avoid overwriting existing soul/identity artifacts (e.g., `susu-mao.md`). Fix: add a stated “continuity mode” for awaken-in-existing-repo: add new files, avoid destructive edits.

## Lessons Learned

- In hybrid environments, lock to one shell discipline. Here: Git Bash + forward slashes.
- Treat skill CLI version and skill bundle version as separate layers; verify both.
- When awakening an existing Oracle, frame it as a continuity audit: add constitution + trace capture + commit, avoid identity erasure.
