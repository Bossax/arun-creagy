## AI Diary
I spent most of this session doing “unsexy” information hygiene: moving files, tightening navigation, and making sure project knowledge is recoverable. The most important move was shifting from one giant global inbox to a two-stage flow: global capture in `ψ/inbox/`, then deterministic routing into a project’s incubate inbox. Once CRDB/CRI/BTR each had `inbox/active/` and `inbox/writing_notes/`, the hub and plan files stopped being abstract intentions and started acting like reliable entrypoints.

CRDB benefited from having a digest + decision log, which made it straightforward to lock Phase 1 choices and keep scope contained. CRI required a more careful interpretation stance: you clarified that Phase 2 should not collapse into a single resilience score but remain multi-scores by capacity type, with asset vs process grouping via the tagging dictionary. That clarification improved alignment between what the project claims to deliver and what the method can defensibly support. BTR was similar structurally: pull triage sources into a project inbox, then make hub/plan point to existing proposal and timeline artifacts rather than rewriting them.

The biggest friction I noticed was link hygiene. VS Code and automation encourage link patterns that are convenient inside one tool but break in Obsidian. Normalizing those links felt repetitive, but it is foundational if the vault is meant to be durable across editors.

## Honest Feedback
1) The work pattern (create inbox, move triage, update hub/plan) repeats across projects and is easy to do inconsistently. Without a single canonical “triage protocol” skill/script, drift is likely: different folder names, incomplete move logs, and inconsistent hub structure.

2) There is constant tension between tool-friendly references (line numbers, editor-specific URLs) and Obsidian-friendly links. If we don’t adopt a strict in-vault rule (“plain relative links only”), the vault silently accumulates broken navigation.

3) Large git churn from earlier migrations remains in the working tree. Even if we avoid committing vault files, that churn increases cognitive load (what changed? what’s safe?) and raises the risk of accidental bundling if a commit is later made.

## Lessons Learned
- Project-scoped incubate inboxes make triage deterministic and keep hub/plan anchored to real inputs.
- Clarification notes are decision evidence: apply them to plan/hub, then store them under project inbox `active/`.
- Link hygiene is a first-class requirement in a dual-editor workflow (VS Code + Obsidian).
