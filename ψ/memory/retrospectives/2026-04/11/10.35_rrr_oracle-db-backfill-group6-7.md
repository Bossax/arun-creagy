## Honest Feedback (friction, ~100+ words)

First, there is cognitive friction in juggling three layers of reality: logs, learnings, and Oracle DB entries. Even with the backfill index, it takes work to keep track of which patterns were added via `oracle_learn()` from logs and which should be treated as canonical learnings in their own right. Second, the supersede wiring is still a manual process. Adding `status` and `superseded_by` across multiple files is error‑prone and would benefit from lightweight tooling or at least a checklist that can be applied mechanically. Third, the asymmetry between vault files and git‑tracked files (especially under `ψ/`) complicates the mental model: I must be strict about not treating vault changes as git commits, even though the editing workflow feels the same inside the editor.

## Lessons Learned

- Before backfilling into Oracle DB, it is worth paying the cost to canonicalise dense learning clusters so that the DB only receives stable, cross‑project patterns instead of a tangle of local variants.
- Supersede metadata (`status`, `superseded_by`) turns older notes from noise into provenance; this supports both human reading and programmatic tooling (e.g. helpers that only consider `status != superseded`).
- For multi-stage reports, treating Chapter 2 and the overall structure as a general pattern makes the CRDB-specific notes more useful as examples instead of one-off rules.
