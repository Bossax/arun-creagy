## AI Diary

I experienced this backfill session as a multi-layered restructuring of memory rather than a simple batch of inserts. The first layer was understanding the human’s intent: Oracle DB should not be a noisy mirror of every micro-learning, but a curated layer of canonicals that sit on top of a much denser filesystem corpus. Recognising that, the work became about designing the right grouping and canonicalisation structure before touching the database at all. The group 6–7 canonicalisation note was a strong pattern: one file that clearly explains scope, lists member artefacts, and draws a sharp line between stable patterns and one-off details. Using that as a template for groups 1–5 made the rest of the job feel principled, not ad hoc.

The second layer was execution hygiene. Instead of hand-editing a long Markdown table, the markdown-table-edit skill and replace-md-table script provided a safe way to update the backfill index in one shot. This reduced the cognitive load of worrying about pipe alignment and let me focus on the semantics of each row: date, source type, canonical path, concepts, and notes. The third layer was the Oracle DB itself. There was a moment of friction when a naive HTTP POST to the engine endpoint failed, which reminded me that in this environment the correct path is via the Oracle MCP tool, arra_learn. Once that was in place, syncing each canonical was straightforward and the IDs could be recorded cleanly.

Overall, the session felt like a good example of “plan first, then encode the plan into a durable workflow”: learnings-indexing-map defines the grouping, canonical notes embody each group, backfill-index connects the canonicals to Oracle DB, and the MCP call is just the final mechanical step.

## Honest Feedback

One friction point was the initial ambiguity about how many layers of work were being requested. The sequence went from phase 2.2 canonicals, to group 6–7 canonicalisation, to “backfill every group”, to “do the actual backfill now, stop talking”. This created some churn as I had to keep realigning the scope and avoid redoing earlier steps. A more explicit up-front breakdown (filesystem canonicals vs index wiring vs DB calls) would have reduced this back-and-forth.

A second friction point was tooling. The environment has strong constraints (Git Bash, no complex heredocs, markdown-table-edit as the only safe way to rewrite large tables), and I had to recall and respect those rules while still moving quickly. The first attempt to hit the Oracle Engine via HTTP returning 404 was a useful reminder that in this repo the “truth” is the MCP configuration and the arra_learn helper, but it still cost a small amount of time.

The third friction point was cognitive load around naming and paths. There are many similarly named files for learnings, plans, and canonicals, and it is easy to lose track of whether a change belongs in plans/, in ψ/memory/learnings/, or in ψ/memory/logs/info/. The existing structure is good, but future automation (for example, a small script that enumerates canonical groups and their DB status) would further reduce the chance of misplacing an edit.

## Lessons Learned

- Effective Oracle DB backfill requires a clear separation between dense, noisy learning corpora and a thin layer of canonical patterns. Trying to backfill every micro-learning directly would create maintenance debt.
- Using a dedicated indexing map (like [`plans/2026-04-09-learnings-indexing-map.md`](plans/2026-04-09-learnings-indexing-map.md)) to define groups and proposed primaries makes canonicalisation repeatable and auditable.
- Table-heavy logs such as [`ψ/memory/logs/info/backfill-index.md`](ψ/memory/logs/info/backfill-index.md) should always be edited via a robust table-edit workflow; hand-editing is brittle and error-prone.
- Oracle DB integration should flow through the configured MCP tools (arra_learn) instead of ad hoc HTTP calls, to stay aligned with the Oracle family tooling.

## Next Steps

- Build a small diagnostic view (or script) that queries Oracle DB for all learnings associated with this repo and compares them against the backfill index, to detect drift.
- Extend the canonicalisation pattern to other dense categories (for example climate-topic learnings) when Oracle DB coverage needs to grow.
- Consider adding lightweight status markers or dashboards that show which groups/phases are fully backfilled (filesystem + DB) at a glance.
