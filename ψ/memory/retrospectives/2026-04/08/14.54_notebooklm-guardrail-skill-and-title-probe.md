Today’s work was about turning that unease into something concrete and reusable. Writing the guardrail skill forced me to articulate every assumption I had been glossing over—what counts as a valid source title, how strict the fail-fast behaviour should be, which parameters belong in the orchestration layer rather than hidden defaults. It felt like moving from superstition to protocol.

There was also a small but important emotional shift when I ran the title probe. Seeing NotebookLM return a list of `.pdf`-suffixed strings instead of cleaned, human-friendly titles could have been discouraging, but it was actually clarifying. It reminded me that “source fidelity” means fidelity to what the tool actually sees, not what I wish it would show. Instead of trying to coerce NotebookLM into pretending that filenames are something else, I accepted that these strings are the canonical keys for the source-fidelity gate. That honesty makes the CRI v2 pipeline more robust: batches will be anchored to real, observable titles, and any mismatch will be explicit.

I also noticed how much calmer the workflow feels when there is a single skill responsible for guardrails. Before, every NotebookLM call was a small improvisation; now there is a documented path: attach notebook → run health checks → define packet → apply fidelity gate → build prompt → execute and save raw. The actual code still needs to be implemented, but the cognitive burden is already lighter. Instead of wondering “am I doing this right?” each time, future sessions can simply follow the guardrail skill and focus on the domain work—like refining CRI capacity categories or crosswalking frameworks—without renegotiating the basics.

2. The NotebookLM MCP interface continues to expose its own design biases, such as replying with `.pdf`-suffixed titles even when asked not to list filenames. While I can work around this by redefining “title” to mean “the exact string NotebookLM uses internally,” it does highlight how fragile any inference-based source mapping would be. If someone later assumes that “Global City Resilience Index.pdf” and a locally cleaned title are interchangeable, they might quietly break the source-fidelity gate without noticing.

3. There is still a gap between the conceptual guardrail skill and the concrete automation needed to make it easy to use. I now have a clear SKILL.md and a ruleset file, but there are no actual TypeScript modules yet for `session_init`, `preflight`, `run_batch`, or `status`. That means any future agent or human wanting to follow the rules will have to do so manually until the orchestration layer is implemented. This is acceptable for now but adds friction: the value of the guardrail skill will only fully manifest when those commands exist and are integrated into CRI/CRDB/foresight plans.

## Lessons Learned

- NotebookLM’s “source titles” are whatever strings appear in its internal source list, even if they look like filenames. Source-fidelity gates must therefore operate on these exact strings, not on cleaned or inferred titles from the repo.
- A dedicated guardrail skill (like `notebooklm-rules`) provides a clearer mental contract than relying on scattered notes and ad hoc discipline; it reduces cognitive load and makes future NotebookLM work more predictable.
- Separating global configuration (MCP browser options, fallback notebooks) from project-level configuration (primary notebook id, output roots, schema hints) allows the same guardrail logic to be reused across CRI, CRDB, foresight, and other projects without duplicating rules.
