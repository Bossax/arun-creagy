
มีคอมเมนต์บนแผนที่ทำมาให้นะครับ สำหรับการจัด Workshop ของ CRBD โครงการของ DCCE ซึ่งเราพยายามจะเอากรอบของ NFCS เข้ามาผูกด้วย เพื่อให้ Workshop ครั้งนี้มี Output บางอย่างที่เอาไปใช้กับ NFCS ได้ แล้วเราก็พูดถึงด้วยว่าตัว NFCS ใน Workshop นี้เราจะไม่ได้พูดออกมาอย่างชัดเจน แต่เราจะพูดว่ามีกรอบนี้เป็นภาพใหญ่ โดยเราจะมาโฟกัสกันที่การสร้างระบบข้อมูลสำหรับ DCCE ซึ่งเป็นระบบข้อมูลที่รองรับบริการชุดข้อมูล (Data Product) เรื่องความเสี่ยงจาก Climate Change การปรับตัว และผลกระทบต่างๆ

ทีนี้ดูจากแผนขั้นต้น Session 1-3 ถือว่าโอเคครับ มีการเฟรมว่าจะใช้ Journey Confirmation เป็น Workflow ซึ่งจริงๆ ลองไป Cross-check กับ Workflow Pattern หรือ Use Case ที่ได้จากการสัมภาษณ์มาด้วยนะว่ามันตรงไหม อาจจะมีคำถามนิดนึงว่า 1-2 Use Case เนี่ย มันอาจจะครอบคลุมได้แค่ 1-2 ภัยหรือเปล่า อย่างเช่น ภัยเรื่องความร้อนกับภัยเรื่องน้ำท่วม แล้วมันจะ Fit กับประสบการณ์หรือ Priority ของคนที่เราเชิญเข้ามาด้วยไหม

เรื่องน้ำท่วมจะเป็นเรื่องที่มีข้อมูลเยอะสุด (Maturity เยอะสุด) เรามีแผนที่และข้อมูลเยอะ ภัยแล้งจะรองลงมา ส่วนใหญ่โฟกัสที่ภาคเกษตร แต่ภัยแล้งในเชิง Consumption หรือ Economic Drought ยังไม่ค่อยพูดถึงเท่าไหร่ ถ้าเราจะเอา Journey นี้มาใส่ ก็จะมีความยากตรงที่จะทำยังไงให้คนมาฟังรู้สึก Contribute หรือเชื่อมโยงได้ อีกเรื่องคือถ้าเอาเรื่องความร้อนเข้ามา ข้อมูลยิ่งจำกัด คนที่เกี่ยวข้องก็อาจจะเป็นกระทรวงสาธารณสุข กรมอนามัย หรือคนที่ทำงานเมือง (อปท., กทม.) ซึ่งจะเป็นวงที่ค่อนข้าง Limit

อาจจะต้องลองดูว่า 1-2 Journey นี้พอไหม สมมติว่าตอนนี้เราแบ่งตามประเภทภัย เราเปลี่ยนมาแบ่งตามประเภทของผู้ใช้ข้อมูลได้ไหม เช่น ผู้ใช้ข้อมูลเชิง Policy ที่ไม่ได้ทำ Impact/Hazard Modeling ต้องลองคิดดูว่าจาก List คนที่จะเชิญมา การออกแบบ Journey แบบนี้เหมาะสมหรือเปล่า

สำหรับ Session 2 กับ 3 ที่ต่อกันเรื่อง Demand Backbone, Service Tier และ Baseline Scale & Access เราสลับกันได้ไหม เอาเรื่อง Baseline ขึ้นมาก่อนแล้วค่อยไปทำ Service Blueprint ให้คนเข้าใจก่อนว่าตอนนี้เรามีอะไรอยู่บ้าง ซึ่งเป็นโอกาสดีให้คนใน Breakout Group เล่าสั้นๆ ว่าตัวเองทำอะไรหลักๆ เพื่อเป็นการ Set Discovery ให้พอเห็นภาพรวมและจำได้ดีขึ้น แต่ไม่เน้นให้มานั่งพูดพรีเซนต์ตัวเองนะ

หลังจากนั้นค่อยมาทำ Service Blueprint โดยมีกระบวนการ Facilitate อาจจะต้องออกแบบตาราง ไดอะแกรม หรือ Visual Guide เพื่อไกด์ว่าต้องเอาอะไรมาเติมตรงไหน มีการจัดหมวดหมู่คำตอบไว้ล่วงหน้าพร้อมตัวอย่าง จะช่วยให้คนเห็นภาพเวลาให้ข้อมูลหรือแสดงความเห็น เราสามารถเอาความเห็นไปจัดใส่กล่องได้ง่าย และเป็นการตีกรอบให้เขาแสดงความเห็นอยู่ในขอบเขตที่เราต้องการ

ส่วน Session 4 เรื่อง Governance Role และ Operating Rhythm เราอาจจะแตะไม่ได้มาก เพราะมันหนักไปทางโครงการ WMO NFCS แล้ว แต่ Workshop ครั้งนี้ต้องโฟกัสที่โครงการของ DCCE เราอาจจะพูดในมุมกว้างๆ ให้เห็นผู้เล่นใน Data Value Chain ว่าองค์กรของเขาอยู่ตำแหน่งไหน (Co-producer, Data provider, User) และ DCCE อยู่ตรงไหน Platform จะ Serve ตรงไหน แต่ถ้าไปถึงระดับ Governance เลยคงยาก เพราะระบบเป็นของ DCCE คนอื่นไม่ได้มี Input เรื่องการดูแลระบบ ดังนั้น Session 4 อาจจะต้องมีการปรับ

Session 5 ก็ปรับตามที่เฟรมมา อาจจะต้องปรับการนำเสนอตอนท้ายว่าเราได้อะไรออกมา มีการย่อยและสรุปเป็น Pattern ส่วนตัว Output Pack Skeleton อาจจะต้องเปลี่ยน ไม่ต้องพูดว่าจะเอา Output ไปใส่โครงการ NFCS แต่แค่พูดว่า NFCS ช่วยให้เห็น Ecosystem ภาพใหญ่ของประเทศยังไง โดยมี DCCE, GISTDA, TMD และอื่นๆ เป็นผู้เล่นหลัก

เรื่อง Workshop Invite & Logistics Package Version 0 มีคอมเมนต์ใส่เข้าไปแล้วเรื่องคนที่น่าจะเชิญมา (เป็นชื่อองค์กรภาษาอังกฤษ) ลองไปเช็กดูว่าการแบ่งกลุ่มตาม Participant Role โอเคไหม เพราะจริงๆ Role มันมีการซ้อนทับกัน เช่น Data Provider หลายคนก็เป็น Processor/Integrator ด้วย หรือ User บางคนก็ต้องเอาข้อมูลไปผลิตต่อเป็น Co-producer ไม่ได้แค่รับข้อมูลไปใช้วางนโยบายอย่างเดียว

ส่วนเรื่องคนที่อยู่ใน Role Governance อย่าง DGA คงมีแค่คนเดียว สมมติเชิญมาแล้วเราจะแบ่งเขาเข้า Breakout Group ยังไง อาจจะเสนอว่ามาสัก 3-4 คนดีไหม ลองดูว่าด้วย Journey และ List องค์กร แต่ละองค์กรควรมากี่คน เป็นไปได้ไหมที่จะอยู่มากกว่า 1 ห้อง แล้วแต่ละห้องจะเป็น Journey เหมือนหรือต่างกัน แต่ถ้า Fix ไว้ก่อนว่ามี 6 ห้อง เพื่อไม่ให้เกินห้องละ 10 คน อันนี้เป็นตัวเลขที่โอเคแล้ว ลอง Revise ดูนะครับ

(OTP) ทำเรื่องแผนขนส่ง เขาอาจจะอยากรู้ว่าใครจะให้ข้อมูลหรือช่วยตีความได้บ้าง ส่วน อปท. หรือ กทม. อาจจะสนใจว่าจะเอาข้อมูลไปใส่แผนเพื่อขอเงินทุนมาปรับเมืองลดความเสี่ยง Climate Change ยังไง ลองปรับ Message ให้ตรงกับ Specific Stakeholder แล้วจัดเป็นกลุ่มใหญ่ๆ ไม่ต้องมีหลายเวอร์ชันเกินไป
